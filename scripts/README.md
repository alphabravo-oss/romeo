# Romeo Scripts

This directory holds small repository automation entry points.

Migration policy: do not generate corrective migration chains during greenfield schema convergence. Keep Drizzle schema changes reviewed in `packages/db/src/schema`, and keep the current baseline candidate at `packages/db/migrations/0000_greenfield_baseline.sql` tight until Phase 19 is accepted and locked. Static baseline review uses `pnpm review:baseline`; pass `--strict` to fail on unresolved lock decisions and `--output path` to write redacted `romeo.greenfield-baseline-review.v1` evidence. Repository conformance coverage review uses `pnpm review:repository-conformance`; it compares the repository contract inventory to the shared conformance test block and writes `romeo.repository-conformance-coverage.v1` evidence. Use its `--strict` mode only when every method is covered or explicitly waived.

Release packaging starts with `pnpm release:pack`. It verifies the workspace, packs `@romeo/api-client` before `@romeo/cli`, checks tarball contents, rejects test files in publish artifacts, and writes `release-manifest.json` with artifact sizes and SHA-256 hashes. Use `node scripts/pack-release.mjs --skip-verify` only after a clean verifier when iterating on package metadata.

Release-channel metadata starts with `pnpm release:channel` after `release:pack` and optional SBOM generation. It reads the release manifest, attaches artifact checksums, compatibility metadata, optional CycloneDX SBOM metadata, and optional release/provenance URLs, then writes `release-channel.json` for update feeds or private release mirrors.

Container scan planning starts with `pnpm containers:scan-plan`. It discovers the Romeo deployment images from Helm values, default CronJob examples, Compose service images, and optional repeated `--image` values, then writes scanner commands plus `release:security` ingestion arguments. Add `--scanner trivy|grype`; add `--cronjob-file path` to replace the default CronJob example set; add `--run` only in environments where the selected scanner is installed and allowed to fetch its vulnerability database.

Release security evidence starts with `pnpm release:security` after `release:pack`, SBOM generation, and optional container scanning. It combines the release manifest, optional CycloneDX SBOM, `pnpm audit --prod --json` output, and optional container scanner JSON files into `security-evidence.json`; pass `--audit-file` and `--container-scan-file` for offline evidence bundles.

Release provenance evidence starts with `pnpm release:provenance` after `release:pack`, `release:channel`, `release:security`, and SBOM generation. It validates manifest/channel/security/SBOM digests, records artifact evidence, hashes source repository/ref/builder/CI metadata instead of returning raw values, and writes `romeo.release-provenance.v1`. Add `--signature-file` or `--signature-ref`, `--attestation-file` or `--attestation-ref`, and the matching `--require-signature` / `--require-attestation` flags in release environments that attach signed provenance. `pnpm release:publish-plan -- --require-provenance` validates this schema and digest linkage before a release can be promoted.

Release approval evidence starts with `pnpm release:approval` after protected release-environment approval has completed. It validates the release manifest and release provenance digests, records only hashed approval references and approver IDs, enforces a configurable minimum unique approver count, checks optional expiration, and writes redacted `romeo.release-approval.v1` evidence without raw approver IDs, raw approval references, secret values, file bodies, or environment dumps. `pnpm release:publish-plan -- --approval-file dist/release/release-approval.json --require-approval` fails closed when approval evidence is missing, blocked, expired, mismatched, under-approved, or redaction-unsafe. This records a local evidence contract; it does not replace the hosted GitHub environment, registry, or change-management approval itself.

Upgrade validation starts with `pnpm release:upgrade-check` after `release:channel`. It checks that a release-channel document has valid target artifacts, SDK-before-CLI publish order, compatibility metadata, and no downgrade from the current version; use `--require-artifact-urls`, `--require-provenance`, and `--require-upgrade-guide` for stricter promotion gates.

Release readback collection starts with `pnpm release:readback-collect` after staging publish. It reads the release manifest, fetches each published package from an npm-compatible registry with `NPM_TOKEN` or `NODE_AUTH_TOKEN`, downloads tarballs with bounded time/size limits, records SHA-256 values, and writes `romeo.release-readback.v1` evidence without token values. Add `--npm-registry-url` for private registries. Use `--readback-plan-file dist/release/release-readback-plan.json` to keep OCI image, Helm chart, Helm repository, and release-asset readback inputs in one reviewed `romeo.release-readback-plan.v1` file; the same plan is accepted by `pnpm release:readback-check` and checked by `pnpm ga:target-preflight`. The plan contains `images[].readback` / `images[].required`, `charts[].readback` / `charts[].required`, `helm.repositoryUrl`, and `assets[].readback` / `assets[].required` entries for `release-channel`, `security-evidence`, `sbom`, `provenance`, and `approval`. The older repeated flags still work: `--image-readback registry/repository:tag@sha256:<digest>`, `--chart-readback name:version[@sha256:<digest>] --helm-repository-url https://charts.example.com/`, and `--asset-readback name=https://releases.example.com/path@sha256:<digest>`. Private OCI, Helm, and release-asset reads use `OCI_REGISTRY_TOKEN`, `HELM_REPOSITORY_TOKEN`, and `RELEASE_ASSET_TOKEN` by default, or `--oci-token-env` / `--helm-token-env` / `--asset-token-env`; HTTP is allowed only for localhost unless explicitly permitted with the insecure flags for a controlled test registry or release-asset fixture. Release readback validation starts with `pnpm release:readback-check`. Use `--planned-readback` in dry-run CI to validate local release manifest, release channel, SBOM, and security evidence wiring. Use `--readback-file path` after credentialed publishing to validate collected package tarball SHA-256 values, release-channel/SBOM/security evidence hashes, verified OCI image digests, verified Helm chart package hashes, and remotely fetched release-asset digests. Live validation requires credentialed npm registry readback plus at least one required image, one required chart, the five required release assets from either `--readback-plan-file` or explicit `--require-image`, `--require-chart`, and `--require-asset` flags, and redaction flags proving tokens, raw registry/readback responses, tarball bodies, OCI manifests, Helm repository bodies, release-asset bodies, and environment dumps were not returned. Local collector/validator regression coverage starts with `pnpm smoke:release:readback`; it stands up a loopback npm/OCI/Helm/release-asset fixture, runs the real collector and validator through the readback plan path, verifies required image/chart/asset checks reject declaration-only entries, and writes `romeo.release-readback-smoke.v1` evidence without retaining token values. Add `--readback-output path` and `--validation-output path` when downstream release/airgap checks need the loopback raw readback and validation files for local contract testing, and use explicit `*-loopback.json` names so the default final-promotion paths remain reserved for target evidence.

Air-gapped bundle verification starts with `pnpm release:airgap-check`. It validates a bundle directory containing the release manifest, package tarballs, release channel, security evidence, CycloneDX SBOM, release provenance, optional release approval, and optional GA bundle, publish plan, raw release readback, or readback-validation evidence. The verifier checks artifact size/hash matches, channel/security/provenance/SBOM version and digest linkage, optional signed-provenance policy, optional approval schema/digest/redaction linkage, optional GA bundle linkage, and optional live readback-validation proof for credentialed package, image, chart, and release-asset readback. Use `--require-readback-validation` with `--readback-validation readback-validation.json` for final disconnected promotion; when `--require-ga-bundle` is also present, the verifier requires the GA bundle to hash-link the same readback-validation file. The generated `romeo.airgap-bundle-verification.v1` evidence omits package contents, SBOM bodies, provenance bodies, approval bodies, raw readback bodies, registry tokens, absolute bundle paths, and secret-like environment values. Use `--require-approval` for final promotion or disconnected transfer packages that must prove protected approval was captured.

Python SDK generation starts with `pnpm sdk:python` while a Romeo API server is available at `http://127.0.0.1:3000`, or pass `--openapi-file` / `--openapi-url` to `scripts/generate-python-sdk.mjs`. The generated package stays under `sdks/python` and uses only the Python standard library.

Python SDK drift checking starts with `pnpm check:sdk-drift`. It exports the in-repo OpenAPI document, regenerates the Python SDK into `tmp/python-sdk-drift-*`, compares it to `sdks/python`, and fails if generated files have drifted. The export uses the CLI workspace `tsx` dependency so the root package does not need another TypeScript runtime dependency.

GitHub CI lives in `.github/workflows/ci.yml`. It wires changed-file formatting, SDK drift, default and opt-in bridge OpenAPI route coverage, typecheck, tests, build, static baseline/repository reviews, branch-protection plan generation, Compose env contract smoke, background job lag smoke, provider resilience smoke, browser automation contract smoke, target quality evidence contract smoke, analytics authorization contract smoke, auth provider acceptance contract smoke, directory sync contract smoke, SCIM lifecycle contract smoke, data connector acceptance contract smoke, tool dispatch acceptance contract smoke, model tool orchestration contract smoke, RAG/vector enterprise contract smoke, voice provider acceptance contract smoke, notification adapter acceptance contract smoke, native client API contract smoke, operational monitoring contract validation, alert firing contract smoke, edge enforcement contract smoke, backup upload-failure smoke, GA checklist draft plus target-preflight and target-plan evidence, live pgvector schema/query-plan/physical-isolation/repository conformance, Helm render smoke, release-evidence dry run with provenance evidence, dry-run release approval evidence, GA evidence bundle generation, air-gapped bundle verification with GA bundle and approval linkage, planned readback collection/validation, loopback release readback smoke, and serialized Compose smoke jobs including local auth/MFA fallback, object-store outage, distributed controls, and tiered RAG. Every Compose smoke matrix entry writes a `dist/ci/*.json` evidence file before artifact upload. `pnpm ci:branch-protection-plan` writes redacted `romeo.branch-protection-plan.v1` evidence for the required GitHub status checks and branch policy posture, and requires the quality job to keep both default `pnpm check:openapi-route-coverage` and `OPENWEBUI_COMPATIBILITY_ENABLED=true pnpm check:openapi-route-coverage` in the command set, keep backend capability contract smokes in the command set, keep `pnpm ga:target-preflight` and `pnpm ga:target-plan` in the command set, and keep the release-evidence job running `pnpm release:approval` plus `pnpm ga:bundle`. `GITHUB_TOKEN="$READ_ONLY_TOKEN" GITHUB_REPOSITORY=owner/repo pnpm ci:hosted-run-verify -- --plan dist/ci/branch-protection-plan.json --output dist/ci/hosted-ci-run-verification.json` reads the hosted GitHub Actions run and jobs APIs, verifies the selected or latest completed workflow run succeeded with every planned required check present and successful, and writes `romeo.hosted-ci-run-verification.v1` evidence without returning job logs, raw API responses, repository slugs, run URLs, or token values. Add `--run-id` or `--head-sha` when tying the evidence to a specific release candidate. `GITHUB_TOKEN="$READ_ONLY_TOKEN" GITHUB_REPOSITORY=owner/repo pnpm ci:branch-protection-verify -- --plan dist/ci/branch-protection-plan.json --output dist/ci/branch-protection-verification.json` reads the hosted GitHub branch-protection API and writes `romeo.branch-protection-verification.v1` evidence for required status checks, review count, stale-review dismissal, code-owner reviews, up-to-date branch requirements, linear history, signed commits, conversation resolution, and admin enforcement without returning the raw API response, repository slug, or token values. Use `--strict` when release automation should fail on blocked hosted settings. Releases that claim hosted CI governance is complete can make it a strict GA gate with `pnpm ga:checklist -- --require-ci-governance-live` or `GA_REQUIRE_CI_GOVERNANCE_LIVE=true`; `pnpm ga:target-preflight` then requires `CI_GOVERNANCE_EVIDENCE_REVIEWED=true`, `GITHUB_REPOSITORY`, one of `GITHUB_TOKEN` or `GH_TOKEN`, and one of `GITHUB_CI_RUN_ID` or `GITHUB_CI_HEAD_SHA` without returning repository or run values. Local regression coverage starts with `pnpm smoke:ci:hosted-run-contract -- --output dist/ci/hosted-ci-run-verification-contract-smoke.json` and `pnpm smoke:ci:branch-protection-contract -- --output dist/ci/branch-protection-verification-contract-smoke.json`. Applying branch-protection settings remains an explicit repository administration step. Credentialed release publishing, protected approval evidence from the release environment, signed/attested provenance attachment, live readback collection, target quality evidence capture, live edge enforcement, and live alert firing are intentionally outside pull-request CI until registry credentials, protected approvals, signing infrastructure, target API, target edge policy, and target monitoring infrastructure are configured.

Compose env contract smoke starts with `pnpm smoke:compose:env-contract`. It compares `deploy/compose/compose.yml` interpolation variables with `deploy/compose/.env.example`, rejects missing or duplicate env-example keys, and writes `romeo.compose-env-contract-smoke.v1` evidence with only variable names/counts and redaction flags. It does not include default values, secrets, Compose bodies, or env-example bodies.

Helm env schema contract smoke starts with `pnpm smoke:helm:env-schema-contract`. It compares `deploy/helm/values.yaml` `.Values.env` keys with `deploy/helm/values.schema.json` env schema properties, rejects missing schema coverage or schema-only env keys, and writes `romeo.helm-env-schema-contract-smoke.v1` evidence with only key names/counts and redaction flags. It does not include Helm values bodies, schema bodies, or secret values.

Scale fixture generation starts with `pnpm fixtures:scale`. It writes synthetic, deterministic fixture data for local, small, or enterprise tiers and validates that fixture string values do not contain common credential patterns. Use `--output path` and `--report-output path` to persist `romeo.scale-fixtures.v1` and `romeo.scale-fixture-report.v1` evidence. Scale load-smoke planning starts with `pnpm smoke:scale:load`, which runs in dry-run mode when no `--base-url` is supplied and validates the driver plan for chat writes, knowledge source writes, run writes, attachment writes, comment/notification writes, knowledge queries, admin listing, local-import connector syncs, and imported OpenAPI tool dispatch-request queue lifecycle. Add `--base-url` plus `--api-key` or `ROMEO_API_KEY` for live API latency evidence against a running Romeo environment. Live tool dispatch-request coverage requires `TOOL_OPERATION_EXECUTION_DRIVER=http-fetch` so the imported connector can become worker-ready, but the smoke queues and cancels dispatch requests without making external worker network calls.

Target quality evidence starts with `pnpm smoke:quality:target`. It reads `/api/v1/admin/analytics/summary`, `/api/v1/admin/analytics/summary.csv`, one or more `/api/v1/agents/{agentId}/eval-release-candidate-evidence` reports, and a representative retrieval replay file through `POST /api/v1/admin/rag/replay` or `/api/v1/admin/rag/replay/compare`. Run it with `--base-url`, `--api-key` or `ROMEO_API_KEY`, repeated `--agent-id` or `--agent-ids`, `--require-eval-passed`, `--replay-file`, and repeated `--forbidden-string` sentinels to persist `romeo.target-quality-evidence.v1` without raw analytics CSV rows, raw agent IDs, raw workspace IDs, eval inputs/outputs, human comments, replay queries, expected chunk IDs, hit IDs, chunk text, provider URLs, credential refs, or secret refs. Eval subject references are recorded as SHA-256 hashes plus presence booleans only. For Qdrant-enabled release evidence, use a compare-shaped replay file and add `--require-vector-comparison --baseline-vector-route-mode pgvector --candidate-vector-route-mode external_vector` or `TARGET_QUALITY_REQUIRE_VECTOR_COMPARISON=true`, `TARGET_QUALITY_BASELINE_VECTOR_ROUTE_MODE=pgvector`, and `TARGET_QUALITY_CANDIDATE_VECTOR_ROUTE_MODE=external_vector`; the evidence then records sanitized route-mode counts and fails if the pgvector baseline or external-vector candidate path was not observed. Operator env equivalents are `TARGET_QUALITY_AGENT_IDS`, `TARGET_QUALITY_REPLAY_FILE`, `TARGET_QUALITY_REQUIRE_EVAL_PASSED=true`, and comma-separated `TARGET_QUALITY_FORBIDDEN_STRINGS`; `pnpm ga:target-preflight` checks the agent list, replay file, and optional vector-comparison shape before reporting the target-quality gate ready. Mount reviewed live evidence through `TARGET_QUALITY_EVIDENCE_PATH` when the running API should expose sanitized readback through `GET /api/v1/admin/target-quality/posture`. Dry-run or no-replay output is planning evidence only and cannot satisfy the GA gate. Local harness regression coverage starts with `pnpm smoke:quality:contract`; it runs the live collector against loopback fixtures and proves analytics, eval, replay redaction leaks, raw eval subject ID rejection, required vector-comparison gaps, and a required failing eval gate are rejected.

It reuses the synthetic scale fixtures and live scale-load driver against an existing Romeo release, requires a non-local `--tier small` or `--tier enterprise`, repeats load runs until both the requested iteration count and `--soak-seconds` are observed, checks namespace and app Deployment readback, records sanitized Deployment/Pod/HPA inventory without raw pod image IDs, stores metadata-only summaries for every live scale-load run, and scans pod logs for the API key plus generated fixture sentinels. Operator env equivalents are `KUBERNETES_LOAD_SOAK_TIER`, `KUBERNETES_LOAD_SOAK_ITERATIONS`, `KUBERNETES_LOAD_SOAK_SOAK_SECONDS`, `KUBERNETES_LOAD_SOAK_INTERVAL_SECONDS`, `KUBERNETES_LOAD_SOAK_TIMEOUT_MS`, and optional namespace/release/service/deployment/base-URL overrides. `pnpm ga:target-preflight` requires a small or enterprise tier, at least two iterations, at least a 60-second requested soak, bounded interval/timeout values, target namespace/app Deployment/app Service posture, and safe base-URL posture when an ingress URL is supplied. Use `--base-url https://romeo.example.com` for an ingress path or omit it to port-forward `--service` from the namespace. Persist GA evidence with `--output dist/ci/kubernetes-load-soak.json`; `--dry-run` writes planned evidence only and cannot satisfy the GA gate. The GA checklist rejects live evidence that omits namespace readback, fixture validation, per-run scale-load summaries, repeated-load, soak-observed, pod-inventory, or pod-log-redaction checks; returns raw `imageID` fields; or lacks log-redaction metadata proving at least one pod was scanned, the scoped API key was checked, and generated fixture sentinels were checked.
